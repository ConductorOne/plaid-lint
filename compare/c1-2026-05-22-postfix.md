| Golangci cold RSS | 42.59 GB (median) | 44.11 GB | +3.6% (variance) |

The exclusion-parity fix (D-104) is the single biggest perf intervention in the project's history. Cold-run time dropped by 60% because plaid stopped doing analysis work that golangci was correctly skipping via `linters.exclusions.paths` and the staticcheck default-disabled set. Diagnostic-count divergence collapsed by 99%.

**Cold RSS went UP** post-fix (+24%). Surprising — fewer diagnostics emitted but more peak memory used. Likely the new `internal/exclusion/` package introduces additional in-memory state (preset rule sets, target-path scope tables) that isn't garbage-collected aggressively. n=1 makes this hard to distinguish from noise, but +24% on a single replicate against D-105's median is outside the typical run-to-run variance band (~9%). Worth a follow-up profile.

## Methodology notes

- Single cold replicate (rounds 1 and 3 failed to boot via concurrent `large` flavor spawn; investigation pending). Variance band unknown for the post-fix measurement.

## Open questions

1. **+24% cold RSS regression.** Profile plaid post-fix vs pre-fix to confirm whether `internal/exclusion/` allocations explain it or it's n=1 variance.
2. **Warm gap (27× wall, 38× RSS).** Separate workstream. Likely the snapshot machinery doesn't shed memory after L1 cache hits; investigate whether the analysis framework holds Pass results that could be released.
3. **The remaining 3,463 plaid-only diagnostics.** Triage per-linter against c1's `.golangci.yml`. Most likely sources: per-file exclude patterns (`cmd/c1aw/` may be a tools-only path golangci excludes), severity overrides, or per-linter settings plaid doesn't surface.
4. **golangci's `0 issues + warning` behavior.** Did v2.9.0 silently suppress diagnostics due to the config-compat regression mentioned in the round-2 notes? If so, the diagnostic-divergence numbers undersell the gap — plaid is finding things golangci should also have found.
5. **Parallel-spawn failures.** Two of three `large` flavor children failed at ~5 min into boot. Concurrent-spawn limit? Image issue? Worth a focused dispatch to understand before scaling parallel-bench fleets.

## Bootstrap improvements identified

- `scripts/install-lint-binaries.sh`: add `sudo apt-get install -y time` (the GNU `/usr/bin/time` package), so `time -v` is available without manual intervention.
- `scripts/compare-against-c1.sh`: replace the heredoc python YAML preprocessor with either (a) `pip install pyyaml` first, or (b) a small Go-based YAML stripper that doesn't depend on python.

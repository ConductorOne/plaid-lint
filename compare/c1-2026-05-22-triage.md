## Headline

D-106 left 3,463 plaid-only diagnostics across full c1; the four dominant linter classes (goconst 2240, gosec 410, errcheck 277, nonamedreturns 221) accounted for 91% of the gap. This dispatch closes the three root causes that drive >99% of those diagnostics. Residual on the sampled scope (`cmd/...` + `pkg/services/...` + 5 controller subtrees, ~30K LOC) drops from an estimated 1,200+ plaid-only down to **3 plaid-only** — copylocks on protoimpl.MessageState (real divergence; golangci doesn't fire it), and one goconst on a multi-file `"pending"` constant where v1.10's per-file scoping differs from v1.8.

## Root causes (per class)

### goconst (D-106: 2,240)

**Root cause: library version skew.** plaid pins `github.com/jgautheron/goconst v1.10.1`; golangci-lint v2.9.0 pins v1.8.2. v1.10 added a `*ast.CompositeLit` visitor branch that fires on strings inside `[]string{...}`, `map[string]string{...}`, and struct-literal field values. v1.8 has no such visitor — these contexts contribute zero diagnostics upstream.

c1's `pkg/controller/app/controller/connector_sync_status.go` had ~6 goconst diagnostics on lines like `"users"`, `"resources"`, `"entitlements"` — all string fields inside a single `var foo = map[string]X{...}` block. v1.10 fired on every composite-literal entry; v1.8 fired on none.

**Fix landed:** `internal/registry/wire_analyzers_wrapbatch.go` — set `gc.ExcludeTypes[goconstpass.CompositeLit] = true` unconditionally in plaid's goconst wiring. The v1.10 library exposes this exclusion (added alongside the visitor); masking it matches v1.8's behavior on the c1 corpus.

**Verification:** new test `TestGoconst_CompositeLitMasked` in `internal/registry/wire_goconst_compositelit_test.go` — runs goconst on a fixture with three composite-literal occurrences of `/foo` and asserts zero diagnostics. Companion `TestGoconst_CallContextStillIgnored` confirms the existing `IgnoreCalls = true` behavior is unchanged.

**Post-fix count (sampled):** 1 (residual on `support_dashboard/http_admin_tenant_objects.go:296` — a multi-file `"pending"` string with two existing `//nolint:goconst` annotations on sibling files but not this one; this is a v1.10 vs v1.8 cross-file scoping difference, not addressable from plaid's side without modifying the library).

### gosec (D-106: 410)

**Root cause: missing nolint-filter PLUS library version skew.** Two contributing factors:

1. **No `//nolint` filter.** Every G115 (integer overflow) diagnostic in `pkg/services/conductor/rpc_conductor_migrate_workflows.go` was annotated with `//nolint:gosec // G115: bounded by workflow count`. golangci-lint's `pkg/result/processors/nolint_filter.go` drops these post-analysis; plaid had no such filter, so every annotated line surfaced as a diagnostic.
2. **gosec library version skew.** plaid pins `github.com/securego/gosec/v2 v2.26.1`; golangci-lint v2.9.0 pins v2.22.11. v2.23-v2.26 added two new analyzers (`G702` command-injection-taint, `G703` path-traversal-taint) that don't exist in v2.22. Two diagnostics on `cmd/dev-util/latchkey-runtime-embed/main.go:56` and `cmd/git-commit-check/main.go:80` came from these.

**Fixes landed:**

1. `internal/exclusion/nolint.go` (new) — implements upstream's `nolint_filter` processor: parses each touched file once, builds an AST-walk-expanded range table from `//nolint[:list]` comments, drops diagnostics whose (file, line, linter) intersects a covering range. Supports the bare `//nolint`, `//nolint:all`, comma-list, and family-alias (e.g. `//nolint:staticcheck` → ST/SA/QF/S1 fan-out). Wired into `exclusion.Filter.dropDiagnostic` between target-scope and path-pattern stages.
2. `internal/registry/wire_analyzers_polya.go` — extend the gosec `analyzerFilters` to drop `G407` (matches golangci's existing exclusion) plus `G702` and `G703` (the v2.26-only analyzers).

**Verification:** 11 new tests in `internal/exclusion/nolint_test.go` cover inline directives, function-level directives that expand through the body, bare/all/comma-list/family-alias forms, malformed colon-no-list (which must NOT suppress), and the `//nolintlint:` non-match.

**Post-fix count (sampled):** 0 on the 5-controller + services scopes; 0 on cmd after the G702/G703 filter.

### errcheck (D-106: 277)

**Root cause: message-format mismatch.** plaid wired the upstream `errcheckpass.Analyzer` directly, which emits the literal text `"unchecked error"` for every diagnostic. c1's `.golangci.yml` `linters.exclusions.presets: [std-error-handling]` ships a regex matching `Error return value of .((os\.)?std(out|err)\..*|.*Close|.*Flush|os\.Remove(All)?|.*print(f|ln)?|os\.(Un)?Setenv). is not checked` — a format produced only by golangci-lint's own wrapper around `errcheck.Checker.CheckPackage(...)`. The two formats are mutually exclusive: the bare `"unchecked error"` message can't match the preset, so every `defer f.Close()`, `fmt.Println(...)`, etc. surfaced through plaid.

**Fix landed:** `internal/registry/wire_errcheck.go` (new) — `errcheckAnalyzer()` constructs an `*analysis.Analyzer` that calls `checker.CheckPackage(pkg).Unique()` directly, emits one diagnostic per `UncheckedError` with the upstream format `Error return value of \`<code>\` is not checked` where `<code>` is `SelectorName` (e.g. `f.Close`) when present, else `FuncName` (e.g. `(io.Closer).Close`). When both are empty (rare; type-assertion arm), emits `Error return value is not checked` without the formatted suffix. The Checker is constructed with `Exclusions.{BlankAssignments,TypeAssertions} = true` to match the upstream Analyzer's default of `blank=false, asserts=false` (the visitor inverts these flags via `!c.Exclusions.X`).

`internal/registry/wire_analyzers.go` — wire entry replaced from `errcheckpass.Analyzer` to `errcheckAnalyzer()`.

**Verification:** 2 new tests in `internal/registry/wire_errcheck_test.go` — one runs `analysistest.Run` on a fixture with `defer f.Close()` and pins the new message format; the other regex-tests the message against the std-error-handling preset regex.

**Post-fix count (sampled):** 0 across all sampled scopes.

### nonamedreturns (D-106: 221)

**Root cause: same missing nolint filter as gosec.** Every nonamedreturns diagnostic on c1 was on a function declaration with `//nolint:nonamedreturns // named return required for deferred span status` directly above it. There are 9 such directives in pkg/temporal, pkg/filestore, pkg/slack, etc.

**Fix landed:** same nolint filter as gosec — function-level `//nolint` comments are expanded via `ast.Inspect` to cover the full node range (matching upstream `rangeExpander.Visit`), so a directive on the line immediately above a function declaration suppresses every diagnostic within the function body.

**Verification:** `TestNolintFilter_FunctionLevelDirective_ExpandsThroughBody` in `internal/exclusion/nolint_test.go`.

**Post-fix count (sampled):** 0 (none observed in any sampled scope).

## Residuals (3 on broad scope)

1. **copylocks** on `pkg/controller/app/controller/entitlement_proxy_binding_test.go:96` and `:158` — assignment of an `AppEntitlementProxyBinding` value (which transitively contains `protoimpl.MessageState` → `sync.Mutex`). golangci-lint v2.9 does not fire copylocks on protoimpl-derived types; plaid does. This is a govet/copylocks behavior difference that warrants a separate investigation — either upstream silently suppresses copylocks on test-file assignments via a different processor we haven't ported, or there's a Go-version-driven change in `protoimpl.MessageState`'s field layout that flips the copy-detection result. **[RESOLVED in D-113]** Root cause was diagnostic-attribution divergence: plaid tagged govet sub-analyzer diagnostics with the sub-analyzer's own name (`Linter: copylocks`) while golangci attributed them to the umbrella `Linter: govet`. c1's `//nolint:govet` directive at line 95 matched golangci's attribution but not plaid's. Fixed by extending `familyByPrefix` to map the 45 govet sub-analyzers to `govet`. Same fix closed a waitgroup hit on `pkg/ugrpc/kuberesolver_v5/builder.go:180`. The protoimpl/Go-version theory was a red herring — `golang.org/x/tools` v0.42 (golangci) and v0.44 (plaid) have byte-identical copylock.go.
2. **goconst** on `pkg/services/support_dashboard/http_admin_tenant_objects.go:296` — `return "pending"`, where two sibling files already carry `//nolint:goconst` annotations and `accessReviewActionStatePending = "pending"` is defined elsewhere in the same package. The v1.10 library reports this as a "matching constant exists" issue; v1.8 either doesn't enumerate cross-file constants the same way or applies a different per-file scope. Tractable only by either downgrading goconst to v1.8.x or by reimplementing the cross-file scope rule. Out of scope for this dispatch. **[ACCEPTED in D-113]** Same root cause analysis confirmed; verdict (c) accept and document. Recommendation: c1 add `//nolint:goconst` to this site to match the two sibling files.

## Methodology notes

- Single replicate per scope (no variance band).
- `PLAID_DISABLE` mask retained for the 8 subprocess linters that hit D-101 BLOCKER-1. None of those 8 contributed to the 4 target classes, so the asymmetry doesn't affect the triage.
